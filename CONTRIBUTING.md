# Contributing to opencloud-registration

Thank you for your interest in contributing! This project relies on developers to remain secure, efficient, and robust.

## Code of Conduct
By participating in this project, you agree to abide by fundamental open-source standards of respect and collaboration. Be considerate and pragmatic in your discussions.

## Development Philosophy
This repository serves as a lightweight, cloud-native companion app to the core OpenCloud ecosystem. We strongly adhere to the following methodologies:

- **12-Factor App Principles:** The application must remain strictly stateless. All configuration must be addressable via Environment Variables (even if file-based parsing via `spf13/viper` is supported).
- **Minimalism:** We rely exclusively on the Go standard library for routing (`net/http`) and templating (`html/template`). Do not introduce heavy third-party frameworks unless strictly necessary.
- **Security-First:** This application handles user credentials. Cryptographic operations (like AES-256-GCM) must be carefully audited. Never log plaintext secrets or PII.

## AI Contributions & Guidelines
We cautiously welcome the use of AI assistants (like Claude, Cursor, or Copilot) to help write code, write tests, and document features. However:

1. **You are Legally and Technically Responsible:** If you submit code generated by an AI, you must be able to comprehend, explain, and maintain it. You are responsible for any license or copyright implications.
2. **Read the AI Agent Rules:** If you are using an AI coding assistant, please ensure it has read and parsed the [`AGENT.md`](AGENT.md) file located in the root of this repository. This file acts as a rigid boundary system to prevent AIs from hallucinating APIs or importing disallowed frameworks.
3. **No Blind Submissions:** Pull Requests consisting of unverified, untested, "spray-and-pray" AI-generated code will be summarily rejected to prevent maintainer fatigue, in line with broader OpenCloud ecosystem quality standards.

## Workflow
1. **Discuss First:** Before starting major work, please open an Issue to discuss the architecture or feature to ensure it aligns with the project roadmap.
2. **Branch Naming:** Use clear, descriptive branches (e.g., `feat/add-phone-field`, `fix/db-locking`).
3. **Testing:** All new functionality *must* include corresponding Go unit tests. See `docs/TESTING_INSTRUCTIONS.md` for our test suites.
4. **Pull Requests:** Ensure your PR description clearly states what the changes do, how they were tested, and links to any relevant Issues.
